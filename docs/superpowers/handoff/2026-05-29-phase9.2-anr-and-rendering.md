## Phase 9.2 작업 (이 세션에서 진행할 것)

### a. LINE batch 렌더링 — ANR 해결의 핵심 ★

**파일**: `app/src/main/java/io/github/june690602_blip/cleancad/render/EntityRenderer.kt`

**전략**:
1. `drawAll()` 첫 번째 forEach 안에서 LINE만 별도 수집 — 색상별로 `HashMap<Int, MutableList<Float>>`에 누적: `[x1, y1, x2, y2]` 4개씩 push
2. forEach 끝난 뒤 색상 그룹마다 `paint.color = key; canvas.drawLines(values.toFloatArray(), paint)` 한 번씩 호출
3. POLYLINE / ARC / CIRCLE / HATCH / ELLIPSE / SPLINE / SOLID / 3DFACE / DIMENSION / LEADER 등은 기존 path-based 그대로 둠 (양 적음)

**Canvas API**:
```
Canvas.drawLines(pts: FloatArray, paint: Paint)
// pts = [x1, y1, x2, y2, x1, y1, x2, y2, ...] — 4개당 1개 line
```

**기대 효과**: LINE이 도면의 ~80%, 색상 종류 100 미만. drawPath 호출 수십만 회 → drawLines 호출 ~100회. **GPU 명령 큐 100배 감소 → ANR 해결**.

**주의**:
- 색상은 `colorFor(entity)` 결과로 그룹화. BYLAYER/BYBLOCK/RGB 모두 colorFor가 final color로 변환해줌.
- 좌표 변환 (`CoordTransform.worldToScreen`) 미리 적용한 screen 좌표를 FloatArray에 push.
- z-order 측면: LINE이 다른 entity보다 앞이든 뒤든 일반적으로 큰 차이 없음. 첫 번째 forEach 끝나면 바로 flush.
- viewport / pix cull은 LINE에도 그대로 적용 (수집 전에 cull).

**의사 코드**:
```kotlin
private val lineBatch = HashMap<Int, MutableList<Float>>(128)

fun drawAll(...) {
    lineBatch.clear()
    entities.forEach { entity ->
        if (entity !is DxfText && entity !is DxfMText) {
            // ... 기존 viewport / pix cull ...
            if (entity is DxfLine) {
                val s = CoordTransform.worldToScreen(entity.start, matrix)
                val e = CoordTransform.worldToScreen(entity.end, matrix)
                val color = colorFor(entity)
                val list = lineBatch.getOrPut(color) { ArrayList(64) }
                list.add(s.x); list.add(s.y); list.add(e.x); list.add(e.y)
            } else {
                draw(entity, canvas, matrix)
            }
        }
    }
    // flush
    for ((color, coords) in lineBatch) {
        linePaint.color = color
        canvas.drawLines(coords.toFloatArray(), linePaint)
    }
    // 두 번째 forEach (text) 그대로
}
```

### b. Outlier viewport 강제 + (선택) 시트 LOD

**전략 b-1**: fit-to-screen 시 viewport를 `displayExtents`로 강제

**파일**: fit-to-screen 로직 위치 찾기 — `app/src/main/java/.../render/DrawingView.kt` 또는 `CoordTransform.kt`. `ViewModel.load` 가 `displayExtents`를 emit하므로 fit-to-screen 계산할 때 그것을 쓰면 됨 ([ui/DrawingViewModel.kt](app/src/main/java/io/github/june690602_blip/cleancad/ui/DrawingViewModel.kt) 또는 [render/DrawingView.kt](app/src/main/java/io/github/june690602_blip/cleancad/render/DrawingView.kt)).

**확인 사항**: 현재 fit-to-screen이 `extents`(raw)에 맞추는지 `displayExtents`에 맞추는지. 로그상 화면 좌측 하단에 outlier가 보이는 걸 보면 **`extents` 기준일 가능성 높음** → 그래서 outlier가 그려짐. `displayExtents` 강제하면 자동 해결.

**기대 효과**: viewport 자체가 displayExtents → EntityRenderer가 `bounds.intersects(viewport)` 컬링으로 outlier entity 자동 제외 → **좌측 하단 검은 덩어리 사라짐**.

**전략 b-2** (선택, b-1만으로 해결 안 되면): 시트 LOD
- `Drawing.sheets` 의 각 시트 bbox를 EntityRenderer에 전달
- 줌아웃 상태(scale × sheetWidth < 50px)에서는 그 시트의 entity 건너뛰고 시트 윤곽선(BoundingBox 사각형)만 그림

### c. 다크모드 선 색상 auto-invert

**파일**: `app/src/main/java/io/github/june690602_blip/cleancad/render/EntityRenderer.kt:71-74` (`colorFor` 함수)

**전략**:
1. `setColors(bgColor, ...)`에서 `bgColor`의 luminance 계산해 저장 (다크 여부)
2. `colorFor(entity)` 결과 색상의 luminance 계산
3. 다크 배경 + 색상 luminance < 0.3 → invert (`Color.rgb(255-r, 255-g, 255-b)`)

**luminance 공식**: `L = (0.299*R + 0.587*G + 0.114*B) / 255.0`

**의사 코드**:
```kotlin
private var isDarkBg: Boolean = false

fun setColors(bgColor: Int, lineColor: Int) {
    this.bgColor = bgColor
    this.defaultLineColor = lineColor
    // ...
    val r = (bgColor shr 16) and 0xFF
    val g = (bgColor shr 8) and 0xFF
    val b = bgColor and 0xFF
    val lum = (0.299 * r + 0.587 * g + 0.114 * b) / 255.0
    isDarkBg = lum < 0.5
}

private fun colorFor(entity: DxfEntity): Int {
    val raw = entityColorByIdentity[entity]
        ?: layerColorMap[entity.layer]
        ?: defaultLineColor
    if (!isDarkBg) return raw
    val r = (raw shr 16) and 0xFF
    val g = (raw shr 8) and 0xFF
    val b = raw and 0xFF
    val lum = (0.299 * r + 0.587 * g + 0.114 * b) / 255.0
    return if (lum < 0.3) Color.rgb(255 - r, 255 - g, 255 - b) else raw
}
```

**검증 시 주의**: 빨강(luminance ≈ 0.3) 같은 중간 명도 색은 invert 안 되도록 임계값 조정. 0.3이 안전.

## 위험 / 회피

- **drawLines 색상 분리**: 단일 paint으로만 그려지므로 색상별 batch 필수. 그룹화 누락 시 모든 선이 한 색.
- **다크 invert 임계값**: 너무 높게(0.5+) 잡으면 빨강/주황 같은 중간 명도 색이 갑자기 invert돼 도면이 이상해짐. **0.3 권장**.
- **viewport=displayExtents 강제**: outlier가 사실 진짜 도면인 경우 일부 사라짐. 이번 도면은 outlier가 노이즈로 확인됐지만 다른 도면은 모름. 향후 "fit to all" / "fit to data" 토글 옵션 검토 필요. Phase 9.2에서는 displayExtents 강제로 가고 회귀 검증.
- **Path 재사용**: drawLines로 바꾼 부분 외에는 매 frame Path 생성하는 부분은 그대로 둠 (POLYLINE 등). 추후 최적화 여지.
