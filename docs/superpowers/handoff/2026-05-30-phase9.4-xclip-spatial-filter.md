## TL;DR — 떡덩어리 근본 원인 확정

**떡덩어리의 진짜 원인은 XCLIP(SPATIAL_FILTER) 미지원이다.** 색 반전·anti-alias·밀도·중복 전부 아니었음(아래 "기각된 가설" 참조).

`04_chamgo.dwg`(= 04_참고도면.dwg, 13.5MB)에는 **SPATIAL_FILTER 객체가 212개** 있다. `elevation crop`, `A$C…`(bound XREF), `standard floor plan` 같은 블록 수십 개가 사실은 "큰 도면의 작게 잘린 창(crop window)"만 보여야 하는데(XCLIP 클립 경계 적용), 우리는 **클립을 무시하고 블록 전체를 렌더**한다. 그 결과 전체 도면(384K~4.8M 크기) 수십 장이 모델공간 한 자리(y≈−530K 근처)에 겹쳐 쌓여 솔리드 떡덩어리가 된다.

ZWCAD는 클립을 적용해 각 crop을 작은 창으로 제자리에 펼쳐 보여주므로 떡이 안 생긴다(다크모드에서도 깔끔, 사용자 확인됨).

→ **Phase 9.4 = XCLIP 지원 구현.** INSERT의 SPATIAL_FILTER를 찾아 클립 경계 밖 블록 엔티티를 컬링/클립한다.

## Phase 9.4 작업 plan — XCLIP 지원

### 목표
INSERT가 SPATIAL_FILTER(XCLIP)를 가지면, 그 클립 경계 밖의 블록 엔티티를 렌더하지 않는다.

### A. INSERT → SPATIAL_FILTER 찾기 (native, dwg_serialize.c `write_insert`)

LibreDWG에 직접 헬퍼 없음. extension dictionary를 수동 순회:
```
INSERT(obj->tio.entity)->xdicobjhandle  (BITCODE_H)
  → resolve to DICTIONARY object
    → entry key "ACAD_FILTER"  → 또 다른 DICTIONARY
      → entry key "SPATIAL"    → SPATIAL_FILTER object
```
- `xdicobjhandle->obj` 로 객체 resolve (NULL 체크 필수).
- DICTIONARY 구조체: `Dwg_Object_DICTIONARY` 의 `numitems`, `texts`(BITCODE_T*), `itemhandles`(BITCODE_H*).
  texts[i] == "ACAD_FILTER" 인 i 의 itemhandles[i] 가 filter dict. 다시 그 안에서 "SPATIAL".
  (DICTIONARY 필드명은 dwg.h에서 `_dwg_object_DICTIONARY` 확인할 것.)
- 헬퍼 시그니처 제안: `static Dwg_Object_SPATIAL_FILTER* insert_xclip(const Dwg_Object *insert_obj);`

### B. SPATIAL_FILTER 구조 (dwg.h ~line 5318)
```c
typedef struct _dwg_object_SPATIAL_FILTER {
  BITCODE_BS num_clip_verts;   // 2 = 직사각형(min,max corner), >2 = 폴리곤
  BITCODE_2RD* clip_verts;     // 클립 경계점
  BITCODE_BE extrusion;
  BITCODE_3BD origin;
  BITCODE_BS display_boundary_on;
  BITCODE_BS front_clip_on; BITCODE_BD front_clip_z;
  BITCODE_BS back_clip_on;   BITCODE_BD back_clip_z;
  BITCODE_BD* inverse_transform;  // 12 doubles (4x3) — clip space ↔ block space
  BITCODE_BD* transform;          // 12 doubles
} Dwg_Object_SPATIAL_FILTER;
```

### C. ⚠️ 좌표계 (가장 주의할 부분 — 정확도 검증 필수)
- `clip_verts` 는 **block 정의 좌표계**에서 정의됨(DXF 스펙). `inverse_transform`/`transform` 이
  block 좌표 ↔ clip 좌표를 매핑.
- 관측: 위 clip_bbox 값들(예 8240,-368591)이 도면 좌표 스케일과 비슷 → block 좌표계가 거의
  WCS와 일치할 가능성. **하지만 추측 금지.** 작은 검증부터:
  1. 진단으로 한 INSERT의 ins_pt + 그 SPATIAL_FILTER clip_verts + 블록 엔티티 좌표를 같이 찍어
     세 값의 좌표계 관계를 실측한 뒤 변환식 확정.
  2. 우리 `write_insert`는 자식 엔티티를 `affine_point(child, new_sx,new_sy,new_rot, child_tx,child_ty)`
     로 world 변환한다. 클립 경계도 **동일한 변환을 거쳐 world bbox**로 만든 뒤 자식 world 좌표와 비교하면
     좌표계 일치 → 컬링 가능.

### D. 구현 단계 (MVP → 완성형)
1. **MVP (떡 즉시 해소):** `write_insert`에서 spatial filter가 있으면 클립 경계를 world bbox로 변환,
   각 자식 엔티티의 대표점(또는 bbox)이 그 밖이면 `write_entity` 호출 스킵. 직사각형(num_clip_verts==2)만
   우선 지원. **경계 걸친 엔티티는 통과(over-draw 약간)시켜도 됨** — 핵심은 멀리 떨어진 대량 엔티티 제거.
2. **검증:** 04_chamgo 로딩 → 떡 영역이 작은 crop 창들로 분리되는지 + extents가 정상(수백K)으로
   줄어드는지 logcat/스크린샷 확인. ZWCAD 화면과 대조.
3. **완성형(후속):** 폴리곤 클립(num_clip_verts>2), 경계 걸친 엔티티 실제 클리핑(라인/폴리라인 잘라내기),
   front/back clip은 2D 뷰어라 무시 가능.

### E. 성능 주의
- 자식마다 클립 bbox 비교는 O(1)이라 부담 없음. spatial filter resolve는 INSERT당 1회만(자식 루프 진입 전).
- 중첩 INSERT: 클립은 해당 INSERT 레벨에서만 적용(자식으로 클립 전파는 복잡 — MVP는 단일 레벨).
